`_fill_advanced_indicators`는 이자부채0 연도의 IC/EV/ROIC/WACC를 **의도적으로 None**(셀트리온형 미포착 가드)으로 둔다. 그러나 `_finalize_company`가 save 직후 호출하는 `_fill_market_cap_and_ev → recompute_and_save_ev_ic`는 이 가드를 모르고 `interest_bearing_debt or 0`으로 **모든 연도 IC/EV를 무조건 계산·저장** → None 가드가 수집 직후 즉시 무효화되고, 매일 `update_all_company_market_caps` 배치가 이를 반복 재발생시켰다. 가드를 단일 진실원(`compute_ic_ev`)에 두어 수집·재계산 양 경로가 같은 정책을 공유하도록 수정.

## 3. ERP 10% → 7% 적용 + 영향 실측
- `config/settings.py` `CALCULATOR_EQUITY_RISK_PREMIUM` 기본값 10.0 → **7.0**.
- **주의(사실)**: `calculate_wacc`가 buffer 0.5%p를 별도 가산하므로 **실효 주식프리미엄은 7.5%**다. 순수 7%를 원하면 ERP=6.5로 둬야 함(사용자 지시대로 일단 7.0 적용).

### 실측 결과 (DB 2683개 평가가능 회사, 분석적 재계산)
| | ERP 10% | ERP 7% |
|---|---|---|
| 2차필터(ROIC−WACC≥2%p) 통과 | **54개 (2%)** | **2584개 (96%)** |
| 평균 spread | −0.16%p | +2.77%p |

방법: WACC 공식상 ERP는 Re에만, Re는 equity_weight 항에만 곱해지므로 `WACC_7 = WACC_10 − equity_weight×0.03`. 저장된 yd.wacc(ERP10 기준)에서 분석적으로 산출. WACC 평균 감소폭 2.93%p ≈ 평균 equity_weight 0.98 × 3%p — 한국기업 자기자본 비중 높음과 일치(검증됨).

## 4. 기술부채 정리
- **classify.py**: docstring의 "5천억원 (5,000,000,000)" — 한글(5×10¹¹)과 숫자(50억)가 100배 모순. 코드·test_filter.py 모두 50억 기준이므로 주석을 코드값(50억)에 맞춤. **값 자체는 안 바꿈**(아래 사람확인 항목).

- 발견: `favorite_detail`의 `int()` 실패 400 분기는 `<int:favorite_id>` 라우터 때문에 **HTTP로 도달 불가능한 죽은 코드**(라우팅 단계 404). 이번엔 그대로 둠.

## LLM이 사용자 대신 내린 설계·결정
1. **코드리뷰 수정 범위 한정**: finder가 찾은 다수 후보 중 #A(확정 버그)+복붙단일화(B)+0이하방어(C)만 수정. 나머지(아래 6번)는 희소/성능/별도리팩터라 보류. → 데이터 정확성 직결 항목 우선.
2. **A 가드 위치**: `recompute_and_save_ev_ic`가 아니라 `compute_ic_ev`(단일 진실원)에 둠. 두 저장 경로가 자동으로 같은 정책 공유 — altitude상 더 깊은 수정. (단 `_fill_advanced_indicators`의 roic/wacc None 처리는 별개라 자체 가드 유지.)
3. **C 방어 추가**: MKTCAP 0이하 제외는 리뷰에서 PLAUSIBLE이었으나 저비용·정확성 직결이라 채택.
4. **ERP는 7로만 적용**: 2차필터 임계값(0.02)·베타(β) 도입은 **건드리지 않음**(사용자 결정사항). 단 96% 통과 문제는 사람확인으로 보고.
5. **classify는 주석만 정정, 값 불변**: 값 변경은 분류 동작을 바꾸는 비가역 결정이라 자율 보류.
6. **favorites 이관 방식**: 테스트 안전망이 0개라 characterization 테스트부터 박제 후 이관. 쓰기 락은 기존에 없었으므로 추가하지 않음(동작 동등 우선).
7. **hook은 최종적으로 제거**: 설정했으나 자동발동 통제 불가 우려로 사용자 요청에 따라 파일·커밋 모두 제거.

## 사람 확인 필요 (자율 보류)
- **classify SMALL_THRESHOLD 값**: 현재 50억(5,000,000,000). docs/지표_사전.md는 "5천억"으로 표기 → 100배 차이. 진짜 의도가 5천억이면 코드가 버그(소/중형 분류가 전부 틀림). 값 변경은 동작/테스트를 바꾸므로 사용자 결정 필요.
- **ERP 7% 후 2차필터 96% 통과**: 필터 변별력이 거의 사라짐(10%는 2%만 통과로 너무 빡빡, 7%는 96%로 너무 헐렁). 임계값(spread) 재조정 또는 베타 도입을 별도 검토 권장.
- **코드리뷰 보류 발견(미수정)**: ①eager `get_stock_code_by_corp_code` O(N×M) 선형스캔(배치는 역맵으로 회피했으나 eager는 미적용, 전체수집 성능) ②`corp_to_stock` 역매핑 비결정성(한 corp_code에 복수 stock_code 시 첫/마지막 불일치) ③`_ensure_krx_index` 빈 dict 캐싱 시 배치 skip 재시도 없음 ④fetch_krx_daily가 stale 스냅샷에도 market_cap_updated_at=now ⑤OFS 폴백이 양쪽 미제출 종목에 DART 콜 2배 ⑥일별 배치 N+1 쓰기. 모두 미수정 — 별도 작업 권고.
