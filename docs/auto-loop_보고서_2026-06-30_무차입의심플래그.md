## 1. 아젠다

**원문**: 모든 연도 이자부채가 0/None인 "무차입 의심(셀트리온형 전수 미포착)" 회사를 잡아내는 플래그/검증 함수를 신설한다 — 입력은 회사의 연간 재무데이터, 출력은 플래그(bool)와 사유이며, 혼합연도·전부0·전부양수·빈데이터·단일연도 경계를 테스트로 박제하고, 2차 필터에서 이런 회사가 근거 없이 통과/탈락하지 않도록 surface한다.

**재진술**: 회사 연간 재무데이터를 받아 "전 연도 이자부채 0/None"이면 `(flag=True, 사유)`를 반환하는 회사 단위 검증 함수를 신설하고, 경계 케이스를 RED 테스트로 박제한 뒤, 2차 필터에서 이런 회사가 근거 없이 통과/탈락하지 않도록 surface(정보 노출)한다.

**핵심 문제**: 기존엔 *연도별* 가드(`compute_ic_ev`: 그 해 이자부채 0/None이면 IC/EV를 None)만 있고 **회사 단위 "전 연도 미포착" 판정**이 없었다. 그래서 무차입 의심 회사가 2차 필터에서 근거 없이 탈락(또는 통과)해도 그 사실이 드러나지 않았다.

## 2. 합격 기준 (박제한 테스트)

- 순수함수 `IndicatorCalculator.flag_no_debt_suspect(yearly_data) -> (bool, str)` (calculator.py)
  - 전부0 / 전부None / 0·None혼합 → `True`
  - 혼합(양수 존재) / 전부양수 → `False`
  - 빈데이터 → `False`(판정 불가)
  - 단일연도 0/None → `True`(사유에 연도수 단서), 단일연도 양수 → `False`
- 2차 필터 surface `CompanyFilter.evaluate_second_filter(corp_code) -> {'passed','no_debt_suspect','reason'}` (filter.py)
  - `passed`는 기존 `check_second_filter`와 **동일**(자동판정 안 바꿈, 정보 노출만)
  - `no_debt_suspect`/`reason`은 무차입 의심 여부 노출

## 5. LLM이 사용자 대신 내린 설계·결정

| DEC-1 | surface = **정보 노출만**(자동판정 안 함) | 사용자 선택. 대안(탈락 보류·별도 '리뷰필요' 상태)은 통과/탈락을 바꿔 기각 |
| 최대반복 | **5회** | 사용자 선택 |
| DEC-2 | 함수 위치=`calculator.py`(이자부채 정책 단일 진실원 `compute_ic_ev` 옆), 시그니처=순수함수 `(yearly_data)->(bool,str)`, 입력=`.interest_bearing_debt` 속성 iterable | DB 없이 순수 판정 → 경계 테스트 용이. 대안(DB 직접조회 함수)은 테스트성↓ |
| DEC-3 | 경계 판정: 전부0/None·혼합(0·None)→True, 양수 1개라도→False, 빈→False, 단일0/None→True(사유에 N=1 신뢰도 단서) | 아젠다 "모든 연도 0/None"의 직역. 단일연도 True 유지하되 신뢰도는 사유로 surface(DEC-1 일관) |
| DEC-4 | 기존 `check_second_filter(->bool)` 불변 + 신규 `evaluate_second_filter(->dict)` 추가 | 회귀 방지. 반환 타입 변경 대신 신규 함수 |
| ~~DEC-5~~ (폐기) | ~~surface 윈도우=valid(roic·wacc not-None) 결정 윈도우~~ | 라운드2 거짓양성 수용으로 도입 → **DEC-6으로 폐기** |
| **DEC-6** | surface 윈도우 = **raw 최근3년 이자부채**(valid 필터 미적용) | orchestrator.py:184-186이 '이자부채 0/None⟹roic/wacc=None'을 강제(roic/wacc 유일 산출 경로). 따라서 valid 윈도우는 부채0 행을 구조적으로 배제 → 무차입 의심을 영원히 못 잡음(기능 사망). raw 최근3년이라야 셀트리온형을 surface. 전 연도 아닌 '최근3년'인 이유: check_second_filter의 `[:3]` 결정 윈도우와 동일 범위(옛 차입연도가 최근 무차입을 가리지 않게) |

**구조적 귀결(검증됨)**: 커플링 때문에 `passed=True & no_debt_suspect=True`는 실데이터에서 불가능하다(무차입이면 roic/wacc None → valid=[] → passed=False). 즉 이 surface가 실제로 잡는 것은 **"무차입 정책 탓에 근거 없이 탈락"**한 회사다. "근거 없는 통과"는 현 파이프라인에서 구조적으로 발생하지 않는다.

## 6. 레드팀 라운드 요약

- **R1**: ① 음수 이자부채 시 사유가 "양수 있어"라고 거짓(`[0,0,-100]`→False인데 사유 거짓) → 사유 문구 교정. ② all-years 윈도우라 최근 무차입을 옛 차입이 가림 → 최근3년 윈도우로(1차 수정).
- **R2**: surface가 valid 결정 윈도우를 모사 안 함 → valid 윈도우로 정렬(2차 수정). **※ 이 공격은 사후 거짓양성으로 판명** — 제시 데이터(부채>0인데 roic=None, 부채0인데 roic=0.10)가 파이프라인 커플링상 불가능. 수용이 기능을 회귀시킴.
- **R3**: ① valid 윈도우 정렬이 커플링 탓에 `no_debt_suspect=True`를 실데이터에서 도달 불가로 만듦(기능 사망) — **치명, R2 회귀 적발** → raw 최근3년으로 되돌림(DEC-6). ② 사유 'N'이 valid수 의미라 오도 → raw 윈도우 복귀로 해소.
